# 动态高度虚拟列表：物理模型可视化指南

如果你觉得代码逻辑太抽象，请看下面的物理图解。我们把虚拟列表想象成一个**“盖楼”**的过程。

---

## 1. 初始阶段：盲猜盖楼 (Initialization)
当我们有 10 万条数据时，浏览器还没画出来，我们不知道每层楼有多高。
我们先统一假设每层楼都是 **50px**。

```text
物理世界 (Phantom 撑高层)
|-------------------|  <-- 楼顶 (Top: 0px)
|   [ 第 #0 层 ]    |  高: 50px, 底边: 50px
|-------------------|
|   [ 第 #1 层 ]    |  高: 50px, 底边: 100px
|-------------------|
|   [ 第 #2 层 ]    |  高: 50px, 底边: 150px
|-------------------|
:       ...         :  (全是 50px 预估高度)
|-------------------|
|   [ 第 #99999 层 ]|  总高度约为: 5,000,000px
|-------------------|  <-- 地基 (Bottom)
```
**此时的状态**：滚动条的长度是基于这 500 万像素生成的。

---

## 2. 测量阶段：多米诺骨牌效应 (Correction)
当你滚动到第 #1 层时，它真正显示在屏幕上了。我们一量发现，它居然有 **80px** 高！

**这就是“多米诺修正”发生的瞬间：**

```text
      修正前 (Before)                 修正后 (After)
|-------------------|          |-------------------|
|   [ #0 ] 50px     |          |   [ #0 ] 50px     | (没变)
|-------------------|          |-------------------|
|   [ #1 ] 50px     |  ----->  |                   |
|-------------------|          |   [ #1 ] 80px     | (变胖了 +30px)
|   [ #2 ] 50px     |          |                   |
|-------------------|          |-------------------| 
|   [ #3 ] 50px     |          |   [ #2 ] 50px     | (被顶下去了 30px)
|-------------------|          |-------------------|
                               |   [ #3 ] 50px     | (跟着一起下沉)
                               |-------------------|
```
**逻辑总结**：
1. 更新 `#1` 的 `height` 为 80，`bottom` 变为 `50 + 80 = 130`。
2. 从 `#2` 开始，所有后续楼层的 `top` 都要等于前一层的 `bottom`。
3. 整个地基（总高度）也随之向下沉了 30px。

---

## 3. 渲染阶段：为什么需要偏移量 (Start Offset)？
这是最难理解的。虚拟列表只有三层结构：

```text
[ 1. Container 外壳 ] -> 固定高度，带滚动条 (窗口)
      |
      |-- [ 2. Phantom 撑高层 ] -> 它是透明的，只负责撑开总高度
      |
      |-- [ 3. List 渲染层 ] -> **它是漂浮的！** (绝对定位)
```

**物理冲突**：当你向下滚动 500px 时，原本在顶部的“渲染层”会跟着滚动条一起滑到上面去，导致你看不见它。
**解决办法**：我们要用 `transform` 把“渲染层”**硬推回来**。

```text
滚动条位置: 500px
=====================  <-- 窗口顶部 (Viewport Top)
          |
          | (这里是空白，因为我们在 500px 处)
          |
   [ 渲染层 (List) ]   <-- 我们把这几条数据通过 translateY(520px) 
   |   #10 数据    |       强行推到用户眼前。
   |   #11 数据    |
   |_______________|
          |
=====================  <-- 窗口底部
```
**偏移量计算公式**：`startOffset = positions[renderStart].top`。
即：渲染层应该定位在“当前渲染的第一条数据”在地图上标注的那个起点位置。

---

## 4. 查找阶段：二分查找逻辑图
为什么不能用除法？因为每层楼高度不一样。

```text
查找目标：用户滚到了 500px，第一条可见数据是谁？

地图表 (Positions):
Index | Bottom
--------------
#0    | 50
#5    | 250
#10   | 530  <-- 目标锁定！
#11   | 610

二分过程：
1. 先看中间 (#50000)，Bottom 太大了，往左找。
2. 不断切半...
3. 最终找到第一个 Bottom > 500 的索引，即 #10。
```

---

## 5. 总结：数据流转全景图

1. **用户滚动** -> 产生 `scrollTop`。
2. **二分查找** -> 在地图里搜寻谁在 `scrollTop` 这个位置。
3. **确定范围** -> 算出 `renderStart` 和 `renderEnd`。
4. **计算偏移** -> 查地图，找出 `positions[renderStart].top`，应用到 CSS。
5. **测量反馈** -> `Item` 渲染后测量真实高度。
6. **修正地图** -> 触发多米诺效应，更新后续所有项的位置。
7. **周而复始** -> 滚动继续，地图越来越准。
